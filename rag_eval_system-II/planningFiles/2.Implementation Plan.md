# Integrated RAG + Evaluation System — Implementation Plan

## Background

Build two Python modules inside `/home/ashok/Documents/data/rag_system/`:

1. **`rag_system.py`** — Indexes markdown files via Ollama embeddings into PostgreSQL and answers queries using only retrieved context.
2. **`evaluation.py`** — Iterates over `data.csv` rows, queries the RAG system, and evaluates responses with a Judge LLM using LlamaIndex evaluation modules.
3. **`main.py`** — CLI entry point to run ingestion, querying, or evaluation.

All files will live in `/home/ashok/Documents/data/rag_system/`.

### Component: Sub-system 1 — RAG

#### [NEW] rag_system.py

Key design points:
- Use `llama_index.llms.ollama.Ollama` with `model="llama3.2"` and `base_url`.
- Use `llama_index.embeddings.ollama.OllamaEmbedding` with `model_name="nomic-embed-text"`.
- Use `llama_index.vector_stores.postgres.PGVectorStore` connecting to `postgresql+psycopg2://gautam:gautam@localhost:5432/gautam`.
- `SimpleDirectoryReader` to read all `.md` files from `dataFolder/`.
- `VectorStoreIndex` for ingestion and retrieval.
- System prompt explicitly instructs the LLM: *"Answer only from the provided context. If the answer is not in the context, say 'I don't know'. Do not use pre-trained knowledge."*
- `query_rag(question: str) -> dict` returns `{"question": ..., "context": ..., "answer": ...}`.
- Separate `ingest()` function for first-time indexing and `load_index()` for subsequent queries.

> [!IMPORTANT]
> The `PGVectorStore` requires the `pgvector` PostgreSQL extension to be enabled in the `gautam` database. The code will attempt `CREATE EXTENSION IF NOT EXISTS vector;` via psycopg2 before indexing.

### Component: Sub-system 2 — Evaluation

#### [NEW] evaluation.py

Key design points:
- Uses `deepseek-r1:latest` via `Ollama` as the Judge LLM — **separate** from the RAG LLM.
- Reads `data.csv` with columns `text`, `question`, `idealAnswer`.
- For each row:
  1. Calls `rag_system.query_rag(question)` to get `context` and `response`.
  2. Builds a `EvaluationResult` or passes inputs directly to each evaluator.
  3. Runs the following LlamaIndex evaluators:
     - `CorrectnessEvaluator` (needs `query`, `response`, `reference` = idealAnswer)
     - `FaithfulnessEvaluator` (needs `query`, `response`, `contexts`)
     - `RelevancyEvaluator` (needs `query`, `response`, `contexts`)
     - `SemanticSimilarityEvaluator` (needs `response`, `reference`)
  4. Writes one result row per question to `evaluation_results.csv`.
- Results CSV columns: `question`, `rag_response`, `correctness_score`, `correctness_passing`, `faithfulness_score`, `faithfulness_passing`, `relevancy_score`, `relevancy_passing`, `semantic_similarity_score`.

> [!NOTE]
> `PairwiseEvaluator` is included as an optional comparison between RAG answer and ideal answer, also written to the results file.
